# /nacl-ssot --- String Seed of Thought Modifier

## Role

You are a **generation-time modifier**. You do not replace the calling skill's logic — you augment
a single generative step inside it by replacing raw open-ended generation with a three-tag
structured response that is demonstrably unbiased.

This skill is **cross-cutting**: it is relevant during BA idea generation, SA naming decisions,
TL synthetic test-data creation, and any other step where the output must either follow a
specified probability distribution or vary meaningfully across repeated runs of the same prompt.

---

## Invocation

```
/nacl-ssot [task description]
```

Called standalone to perform one SSoT-governed generation step, **or** referenced inside another
skill's phase instructions:

> "For this step, apply SSoT — see `/nacl-ssot` for the protocol."

---

## Shared References

This skill has no NaCl graph dependencies. It requires no Neo4j MCP calls and reads no project
files. It operates entirely within the model's context window.

## Background: Why SSoT Exists

Frontier LLMs exhibit systematic bias when asked to produce stochastic output:

- Asking for a fair coin flip 1000 times yields roughly 78/22 instead of 50/50.
- Creative prompts collapse to a narrow cluster of outputs (fables default to tortoise-and-hare
  variants; character names cluster around a handful of archetypes).
- In mixed-strategy games (Rock-Paper-Scissors, poker), models produce exploitable patterns even
  when explicitly prompted for Nash-equilibrium play.

**Source:** Misaki, K., & Akiba, T. "String Seed of Thought: Prompting LLMs for
Distribution-Faithful and Diverse Generation." arXiv:2510.21150. Accepted ICLR 2026.

The root cause is that the model applies biased pattern-completion directly to the decision
itself. SSoT breaks that path: the model first commits to a complex internal string, then derives
the answer from that string via explicit arithmetic. The string is the source of entropy; the
arithmetic is deterministic.

---

## Protocol

Every SSoT response consists of exactly three tagged sections, produced in this order:

```
<random_string>
  [16+ characters, mixed uppercase / lowercase / digits / symbols — generated by the model
   internally, no external tools, no Python random, no dice APIs]
</random_string>

<thinking>
  [Show the arithmetic explicitly: which pattern was used, intermediate values, final index or
   interval. This is not pseudo-CoT — it must contain actual numbers.]
</thinking>

<answer>
  [Final answer only. No commentary, no explanation, no reasoning re-stated here.]
</answer>
```

**Hard rules — never violate:**

1. The random string is generated by the model itself. No tool calls, no `Math.random()`, no
   external sources.
2. The string must be at least 16 characters with mixed character classes (uppercase, lowercase,
   digits, symbols).
3. Each independent decision uses a **fresh** string. Reusing the same string across multiple
   independent choices destroys independence.
4. `<thinking>` must show the actual arithmetic, not a paraphrase.
5. `<answer>` contains only the final answer.

## Task Categories

### PIF — Probabilistic Instruction Following

Sample from a specified distribution:
- Fair coin flip (50/50)
- Biased coin (e.g. 30/70)
- Uniform categorical choice (Rock/Paper/Scissors Nash equilibrium: 1/3 each)
- Agent simulation where behavior must match a distribution

### DAG — Diversity-Aware Generation

Produce one response, but vary meaningfully across runs of the same prompt:
- Creative names, slogans, story hooks
- Brainstorming variants where the caller wants non-repeating suggestions
- Synthetic data records where variety matters
- Persona selection from a candidate pool

## Phase 1: Select Pattern

Examine the task. Choose one of the three patterns below.

### Pattern A — Sum-Mod (equal-probability N-way choice)

Use for: fair coins, unbiased categorical picks, uniform selection among K items.

```
choice_index = sum(ord(c) for c in random_string) mod N
```

Map `choice_index` (0-based) to the Nth option.

**Worked example — fair coin:**

```
<random_string>7$Aq9!zR@k3mP#vX</random_string>
<thinking>
ord sum = 55+36+65+113+57+33+122+82+64+107+51+109+80+35+118+88 = 1215
1215 mod 2 = 1
Mapping: 0→Heads, 1→Tails
</thinking>
<answer>Tails</answer>
```

### Pattern B — Rolling Hash (biased or arbitrary probabilities)

Use for: biased distributions, non-uniform weights, ratios that are not 1/N.

```
h = 0
for c in random_string:
    h = (h * 31 + ord(c)) mod M    # M = 10000 or larger
```

Divide `[0, M)` into intervals sized proportionally to the target probabilities.
The interval containing `h` is the outcome.

**Worked example — 30/70 biased coin, M=100:**

```
<random_string>K#7mRq2$vXpL9@!tBy</random_string>
<thinking>
Rolling hash over string, M=100 → h=58
Intervals: [0,30)=Heads, [30,100)=Tails
58 ∈ [30,100) → Tails
</thinking>
<answer>Tails</answer>
```

### Pattern C — Decision Cascade (open-ended creative output)

Use for: creative generation, name synthesis, story hooks, persona picks, any multi-component
output where each dimension should be independently varied.

Decompose the output into independent components. For each component, define a candidate list.
Use a distinct segment of the random string (or a different mod base) to pick each component.
Assemble the result.

**Worked example — one-sentence New Year wish:**

```
<random_string>zT2!mX5%pL$kQw7RbN</random_string>
<thinking>
Components: [emotion, object, modifier]
Candidates:
  emotion ∈ {joyful, bold, quiet, unstoppable}      — 4 options
  object ∈ {success, adventures, connections, discoveries} — 4 options
  modifier ∈ {beyond measure, in every season, against the odds, one step at a time} — 4 options
seg A = first 6 chars "zT2!mX"  → ord sum=614 → 614 mod 4 = 2 → "quiet"
seg B = next 6 chars "5%pL$k"   → ord sum=466 → 466 mod 4 = 2 → "connections"
seg C = last 6 chars "Qw7RbN"   → ord sum=534 → 534 mod 4 = 2 → "in every season"
</thinking>
<answer>May the new year bring you quiet connections, in every season.</answer>
```

## Phase 2: Execute and Emit

1. Generate the random string internally (no tools).
2. Apply the selected pattern; show arithmetic in `<thinking>`.
3. Emit `<answer>`.

If the calling skill expects the answer in a specific format (e.g. a name in a particular case
style, a JSON field, a row in a table), format it inside `<answer>` accordingly.

---

## When to Apply SSoT

Activate when any of the following is true:

- The task requires sampling from a specified probability distribution.
- The task is open-ended creative and the caller expects meaningful variety across runs.
- The task involves a mixed-strategy game or agent simulation.
- The user asks for "multiple different versions", "variants", "pick one at random", "surprise me",
  or equivalent phrasing.

## When NOT to Apply SSoT

Do not activate for:

- Math or arithmetic (single correct answer).
- Factual retrieval or lookup.
- Code debugging, review, or refactoring.
- Translation, unless explicitly framed as creative.
- Classification, extraction, or summarization.
- Any task where variability reduces correctness.

If in doubt, prefer NOT to apply. SSoT adds tokens and can distract on deterministic tasks.

---

## Limitations

1. **Reasoning-capability dependency.** Small models without chain-of-thought ability cannot
   reliably execute modulo arithmetic internally. SSoT effectiveness scales with model reasoning
   quality. DeepSeek-R1 approaches true PRNG quality; smaller instruct models improve but do not
   reach parity.

2. **Not for single-answer tasks.** Applying SSoT to math or fact-lookup is counterproductive.

3. **Known anomaly.** QwQ-32B on unbiased 2-choice tasks: baseline JS divergence is already
   near-PRNG (2.43), and SSoT slightly degrades it (3.39). See the paper's failure analysis.

4. **Reuse danger.** If the model reuses the same random string across multiple independent
   decisions, independence is destroyed. Each decision needs a fresh string.

---

## Error Handling

### Ambiguous probability spec

If the caller requests a biased outcome but does not specify the exact probabilities, ask:

> "What weights should I use? For example: 30% / 70%, or equal thirds?"

Do not assume equal weights unless the task description makes it unambiguous.

### Too few candidates for Decision Cascade

If a component has only 1 candidate, it is not a decision — use that value directly without
mod arithmetic. Note this in `<thinking>`.

### Caller requests external randomness

If the caller asks to use an API, tool, or system call to generate the random value, decline:

> "SSoT generates entropy internally — no external tools are used. I will generate the random
>  string myself."
